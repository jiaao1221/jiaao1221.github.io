Spring Security 认证原理(二)

发表于 2019-01-03 | 更新于 2019-01-04 | 分类于 学习笔记 | 阅读次数：

开始

上一篇主要介绍了凭证以及session存取登陆状态，利用 SecurityContextHolder持有安全上下文，以供之后使用。这一篇主要介绍验证器和鉴权器。

验证器

凭证已经有了，但还需要验证其是否合法，但验证方式有各种各样，例如用户密码，手机验证码，人脸识别等，为了能使用不同的验证器，我们可以设计一个验证管理器，其包含了不同的验证器，通过判断凭证是否适配当前的验证器选择适当的验证器进行验证。

AuthenticationManager就是验证管理器，providers字段就是其包含的验证器list。验证方法是由authenticate函数实现，如果provider.supports(Authentication)返回true，则使用该provider进行验证。如果遍历完所有的provider后result仍为空，并且父AuthenticationManager不为空，则使用父管理器进行验证。
由此可见，AuthenticationProvider才是验证器的实现：

在spring security中，默认使用的是继承AbstractUserDetailsAuthenticationProvider的DaoAuthenticationProvider。
先看看AbstractUserDetailsAuthenticationProvider中的关键方法：

首先通过用户名在缓存里找该用户，如果未找到，则调用retrieveUser()方法找到该用户的信息，这个信息是用户注册时记录的。preAuthenticationChecks()做一个前置检查，判断账户是否被锁定，是否能使用，是否过期。additionalAuthenticationChecks()方法才是真正做检验的地方。retrieveUser和additionalAuthenticationChecks都是由子类DaoAuthenticationProvider实现

在retrieveUser方法中，是利用UserDetailsService来获取用户信息的。我们可以自己实现这个类，来达到使用数据库或者redis存取用户信息。additionalAuthenticationChecks方法则是使用passwordEncoder来判断用户的口令是否匹配，这个passwordEncoder实际上就是个加密类，我们可以自己实现或者使用默认。
整个逻辑也很简单，除去异常情况的判断外，就是在数据库或者内存中通过用户名找到该用户的信息，并检查请求提交的口令是否和这个匹配。如果有其他的验证方式，可以自己实现provider并注册进去。

鉴权器

终于到最后了，我们已经得到认证了的凭证，其中包含了用户拥有的权限，最后我们需要比较申请资源需要的权限和用户权限，如果匹配的话就放行，否则返会accessdeny。
用户的权限在凭证就可以得到，但资源的权限还没有，在spring security中抽象为SecurityMetadataSource：

这里就不介绍实现类了，实现的方式有很多，最简单的就是用map存储url对应的权限。也可以自己实现由数据库存取的SecurityMetadataSource。
AccessDecisionManager负责验证用户是否有操作权限。Spring Security引入了投票器的概念，真正的授权功能是通过一组AccessDecisionVoter来实现的。
AccessDecisionManager维护着一个AccessDecisionVoter列表参与授权的投票。根据处理投票的策略不同
Spring Security中AccessDecisionManager有3个不同的实现。
（1）UnanimousBased（全票通过）：所有投票器都通过才允许访问资源。
（2）ConsensusBased（少数服从多数）：超过一半的投票器通过才允许访问资源。
（3）AffirmativeBased（一票通过）：只要有一个投票器投票通过，就允许访问资源。

vote()方法的返回结果会是AccessDecisionVoter中定义的三个常量之一。ACCESS_GRANTED表示同意，ACCESS_DENIED表示返回，ACCESS_ABSTAIN表示弃权。如果一个AccessDecisionVoter不能判定当前Authentication是否拥有访问对应受保护对象的权限，则其vote()方法的返回值应当为弃权ACCESS_ABSTAIN。
投票器的实现在这里先不讨论，之后会涉及到。
